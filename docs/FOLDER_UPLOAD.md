为什么合理

- 有真实场景:Agent 工具被 jail 在 session workspace 里(WorkspacePaths.java:51-68),用户想让 Agent
  分析一个项目/一批文档时,目前只能一个个挑文件,目录上传是自然的补全。
- 下游已就绪:文件列表用 Files.walk 递归遍历(AgentFileService.java:44-60),前端 FileTreeView 已能渲染嵌套路径,Docker sandbox 把整个
  agent 目录挂载为 /workspace。也就是说消费端天然支持子目录,只是上传端进不去。
- 改动面小:浏览器原生支持 webkitdirectory,后端 multipart 已支持多文件,主要是打通"相对路径"这一环。

主要障碍(需要改的地方)

1. 后端故意拍平路径 —
   AgentFileService.saveUploads(AgentFileService.java:104-107)只取文件名最后一段,目录结构会被丢弃,同名文件互相覆盖(如 a/index.js 和
   b/index.js)。需要放开这个拍平,但保留已有的 ../越界防护(108-114 行)。
2. 前端只传文件名 — uploadAgentFiles(api.ts:1267)用 f.name 作为 part 的 filename,需要改为
   webkitRelativePath(目录选择时浏览器会填充)。
3. 附件面包屑 — 发送消息时前端拼 [Attached: /workspace/<name>](chat-screen.tsx:1389),目录上传时逐文件列会刷屏,应改为类似 [Attached
   directory: /workspace/<dirname>/ (N files)]。

我的建议方案(概要)

- 后端:在现有 POST /api/agents/{agentId}/files 上支持 part filename 中的相对路径(规范化 + 复用越界防护),不必新开 endpoint;拍平行为对普通附件仍然成立(单文件 filename 本来就没路径)。
- 前端:composer 附件按钮旁加"上传文件夹"入口(webkitdirectory),带忽略规则、数量/体积预检、分批上传和进度提示。

需求本身没有问题,核心是做好防呆(node_modules 类目录)和分批上传。如果你认可这个方向,我可以进入 plan 模式细化实现方案。
